# 2026.05.26 — YonerAI v0.1.0-alpha.3 Real Hybrid Execution Slice

Release date: 2026-05-26
Tag: `v0.1.0-alpha.3`
Status: alpha pre-release
Previous pre-release: `v0.1.0-alpha.2`

## Summary

YonerAI v0.1.0-alpha.3 adds a public-safe, local-dev Hybrid execution slice
that users can run from the CLI. It connects route preview, a verified test
Local Node session, in-memory loopback relay transport, mock provider execution,
redacted run events, and deterministic Oracle stub request/result envelopes into
one report with run IDs. This is still alpha software: it is not production
Oracle, not Official Managed Cloud runtime, not live Discord, not a public
tunnel, and not production-ready.

## What you can actually run now

| Command | What it does | What it does not do | Mode |
| --- | --- | --- | --- |
| `yonerai hybrid run --pretty` | Runs the local-dev Hybrid slice end to end and prints a readable report: selected route, Local Node relay status, provider run ID, Oracle stub run ID, route matrix, and explicit non-actions. | Does not contact production Oracle, Official Managed Cloud, live Discord, public tunnels, or external providers by default. | Local-dev, in-process/loopback |
| `yonerai hybrid run --json` | Emits the stable `yonerai-hybrid-execution-slice/v0.1` report for tests or automation. | Does not persist message bodies or expose provider keys. | Local-dev JSON contract |
| `yonerai node status --json` | Shows the fixture Local Node capability manifest, heartbeat, session reference, posture, and hash-only session-token status. | Does not start a real node process or expose a public endpoint. | Local-dev fixture |
| `yonerai node pair --dry-run --json` | Shows the one-time pairing/orchestration envelope path without pairing a real node. | Does not create production trust, production signing keys, or a live control-plane session. | Dry-run fixture |
| `yonerai oracle status --json` | Shows that the deterministic Oracle stub queue is locally available. | Does not use production Oracle or Official Managed Cloud. | Local-dev stub |
| `yonerai oracle queue --json` | Enqueues a public cloud-contract candidate into the local-dev Oracle stub and returns a request/result envelope with a run ID. | Does not send raw private content, raw prompt bodies, provider keys, or network calls. | Local-dev stub |
| `yonerai route preview "hard public reasoning over public API docs" --json` | Shows route/contract decisions and why official cloud remains contract-only in the public repo. | Does not execute provider calls, shell commands, file reads, or production routing. | Preview-only |
| `yonerai start --guided --lang ja --pretty` | Explains the first five minutes in plain language, including the new Hybrid local-dev command. | Does not install, deploy, mutate PATH, or send prompts to a model. | Local diagnostic |
| `yonerai demo --pretty` | Shows the current public demo surface, including the real Hybrid execution slice check. | Does not require credentials or live services. | Mock/offline demo |
| `yonerai doctor --json` | Reports local CLI, manifest, provider, node/relay, Oracle stub, and Hybrid execution-slice status. | Does not perform installation, production health checks, or live service calls. | Offline diagnostic |

## Hybrid node, relay, and Oracle

The public repository now includes a local-dev Hybrid execution path that can be
run without credentials. The Local Node side is represented by test fixtures:
capability manifest, heartbeat, one-time pairing, session reference, posture,
and hash-only session token evidence. The relay transport is in-memory and
loopback-only. The Oracle side is a deterministic local-dev stub queue that
returns request/result envelopes and run IDs.

This is useful for exercising the public contract and safety boundaries. It is
not a production Oracle implementation, not a private control plane, and not a
hosted cloud service.

## Routing

The route matrix now demonstrates four public-safe decisions:

- `local_preferred` for local/public documentation work.
- `hybrid` for private workspace work that must stay local.
- `cloud_contract_candidate` for hard public reasoning that can be represented
  by an Oracle stub envelope.
- `deny` for dangerous operations when required capability or approval is not
  satisfied.

Private file content, provider keys, and raw prompt bodies are not sent to the
Oracle stub/cloud candidate path.

## Provider runtime

The default execution path remains mock provider execution. `yonerai hybrid run`
uses the mock provider by default and returns a provider `run_id`. The local LLM
provider remains loopback-only and requires explicit live opt-in. External
providers remain off by default and require explicit provider-specific env
configuration and `--live`.

## Demo, doctor, and first run

`yonerai demo`, `yonerai doctor`, and `yonerai start --guided` now surface the
Hybrid local-dev slice in plain language. They distinguish what is real
locally, what is a stub, and what remains outside the public repo.

## Security and boundaries

- No production Oracle or Official Managed Cloud runtime is included.
- No production signing key or production trust store is included.
- No deploy, public tunnel, or live Discord connection is included.
- No provider keys are required for default tests or default demo paths.
- Live external provider calls remain opt-in only.
- Local LLM endpoints are loopback-only.
- Local Node sessions are represented with hash-only token evidence.
- Relay/audit fixtures do not persist message bodies.
- Workspace/private content does not enter Oracle stub payloads.
- Dangerous operations are denied or approval-gated; arbitrary shell/file/tool
  execution is not enabled.
- `src/cogs/ora.py` remains a legacy boundary and is not claimed solved.
- `reference_clawdbot` was not modified.

## Known limitations

- Not production-ready.
- Official Managed Cloud is not runnable from this public repository.
- Oracle/control-plane behavior is a local-dev stub/contract only.
- Live Discord is not restored.
- Persistent memory is not complete.
- Live web search is not default/live.
- SafeShell is not arbitrary shell.
- Workspace file access is an allowlist guard, not PDF/image parsing, folder
  crawling, arbitrary file access, or real LLM summarization.
- Production installer, npm, and winget are not ready.
- No Google login.
- No production DB behavior.
- No production telemetry ingestion.
