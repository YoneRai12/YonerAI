## Summary

YonerAI v0.18.0-alpha.2 publishes public-safe evidence for the verified staging
Google auth E2E path and hardens the public CLI staging bridge against
token-like URL leaks.

The staging path is:

1. CLI starts a one-time bridge request against `https://api-staging.yonerai.com`.
2. Browser reaches the staging Google callback.
3. Backend callback marks the bridge request linked.
4. CLI polls linked state and verifies `GET /v1/account/me`.
5. CLI stores only a redacted YonerAI staging account claim.

This is staging-only. It is not production Google login, not production
AWS/Oracle runtime, and not account sync.

## What Users Can Run

Set the staging origin:

```powershell
$env:YONERAI_STAGING_AUTH_ORIGIN = "https://api-staging.yonerai.com"
```

Run staging login:

```powershell
yonerai auth google login --staging --bridge --open-browser --wait-linked --pretty --lang ja
```

Check the linked state:

```powershell
yonerai auth status --pretty --lang ja
```

Open the TUI auth screen:

```text
/認証
/auth
```

The command may open a browser when `--open-browser` is explicit. In non-TTY or
browser-fallback environments, the CLI prints a staging browser URL instead.
That URL is not a Google access token, refresh token, or auth code.

## E2E Evidence

Public-safe evidence file:

- `docs/evidence/STAGING_GOOGLE_LOGIN_E2E.md`

Verified staging behavior:

- `GET https://api-staging.yonerai.com/v1/status` returned `200`.
- `GET https://api-staging.yonerai.com/v1/rate-limit` returned `200`.
- unauthenticated `GET /v1/account/me` returned controlled `401`.
- `POST /auth/cli/start` created a one-time bridge request.
- browser staging Google callback was reached.
- bridge poll reached linked state.
- authenticated `GET /v1/account/me` returned `200` with minimal profile.
- public CLI did not receive or store Google access tokens, ID tokens, refresh
  tokens, or auth codes.
- refresh token persistence remains disabled.

No account email, request id, auth code, token value, client secret, provider
key, private path, or private runtime inventory is included in the public
evidence.

## Security and Privacy Boundary

- staging origins are allowlisted
- non-HTTPS public origins are rejected
- credentialed URLs and private IP origins are rejected
- bridge redirects are rejected before following them
- bridge paths containing backslashes, absolute URLs, token-like query
  parameters, or token-like URL fragments are rejected
- bridge `expires_at` must be a public scalar value
- nested credential-like JSON keys are rejected
- Google client secrets are not required or supported in the public CLI
- Google token exchange is not performed in the public CLI
- token values are not printed
- refresh token plaintext storage is not enabled
- staging session token values are not stored
- local private memory, local file content, and local node content are not
  uploaded
- OpenAI shared traffic remains off
- production Google login remains unavailable

## Known Limitations

- This is an alpha prerelease line, not the stable CLI Local Runtime line.
- Production Google login/passkey/account runtime is not included.
- Production AWS backend is not included.
- Production Oracle/cloud runtime is not included.
- Account sync is not performed by the public CLI.
- Live Discord is not restored.
- OpenAI shared traffic is not enabled.
- Local private memory is not uploaded automatically.
- Production cloud memory sync is not included.
- Production installer signing/trust store is not included.
- npm, winget, and auto-update apply are not included.
- Arbitrary shell/file/tool execution is not enabled.
- `src/cogs/ora.py` remains a legacy boundary and is not solved by this release.
- `reference_clawdbot` is not touched.
