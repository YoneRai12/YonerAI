## Summary

YonerAI v0.21.0-alpha.1 makes the staging Control Spine CLI easier to use
without changing the production boundary. Normal users can start with short,
safe commands such as `yonerai login`, `yonerai whoami`, `yonerai sessions`,
`yonerai projects`, `yonerai ping`, and `yonerai rate-limit`.

This is staging Control Spine only. It is not production Google login, not
production Official Managed Cloud, not production Oracle/cloud runtime, and not
automatic local-to-cloud upload.

## Runnable Changes

- `yonerai login` is now the normal staging login path.
- Expert flags such as `--bridge`, `--open-browser`, `--wait-linked`,
  `--pretty`, and `--lang ja` remain available for CI or advanced testing, but
  they are hidden from normal help output.
- Short aliases are available for account, session, project, API ping, and
  rate-limit checks.
- Japanese command display modes are configurable:
  - `ja_only`
  - `ja_with_en`
  - `en_with_ja`
  - `en_only`
- Secondary command aliases are dimmed in the interactive palette when color is
  enabled.
- `auth status` and Control Spine screens show staging session expiry and the
  next safe re-login action.
- Expired or missing sessions produce a controlled Japanese message and suggest
  `yonerai login`; stack traces are not shown.
- The CLI reads staging `/v1/health` `api_version` and `min_cli_version` when
  present, and warns politely only when the backend requires a newer CLI.
- Missing `/v1/health` version fields are debug-only and do not nag users.
- `agent:run` and `admin:*` scopes are frozen by default and marked as requiring
  a separate threat-model gate.
- `CURRENT_TRUTH.md` is added as the public anchor that future AI lanes should
  read before making release, API, CLI, or production-readiness claims.

## Commands

Set the staging origin when using the staging backend:

```powershell
$env:YONERAI_STAGING_AUTH_ORIGIN = "https://api-staging.yonerai.com"
```

Short safe commands:

```powershell
yonerai login
yonerai whoami
yonerai sessions
yonerai revoke <session_id>
yonerai logout
yonerai projects
yonerai projects current
yonerai projects use <project_id>
yonerai ping
yonerai rate-limit
```

Command display settings:

```powershell
yonerai config set command_display ja_only
yonerai config set command_display ja_with_en
yonerai config set command_display en_with_ja
yonerai config set command_display en_only
```

Staging sync shortcuts remain explicit:

```powershell
yonerai sync status --pretty --lang ja
yonerai sync conversations --pretty --lang ja
yonerai sync preview cloud-to-local --pretty --lang ja
yonerai sync preview local-to-cloud --pretty --lang ja
```

## Security And Privacy Boundary

- Production Google login remains disabled.
- `login --no-staging` remains rejected.
- Production login acceptance must land only through
  `docs/PRODUCTION_GRADUATION_GATE.md` in a separate future PR after Google
  verification passes.
- The public CLI must not store Google access tokens, Google ID tokens, Google
  auth codes, or refresh tokens.
- The token-custody regression is now a standing Quality Wall test.
- OpenAI shared traffic remains disabled.
- Local private file, memory, and local node content are not uploaded
  automatically.
- No arbitrary shell/file/tool execution is enabled.
- No production Oracle/cloud runtime is included.
- No provider keys are included, printed, or stored.

## Known Limitations

- This is an alpha prerelease line, not the stable CLI Local Runtime line.
- Production Google login/passkey/account runtime is not included.
- Production AWS backend code is not included in the public repository.
- Production Oracle/cloud runtime is not included.
- Official Managed Cloud remains external/private and contract-only from the
  public repository.
- Cloud conversation sync remains staging preview only.
- Local-to-cloud upload is not performed automatically.
- Production installer signing/trust store is not included.
- npm, winget, and auto-update apply are not included.
- `src/cogs/ora.py` remains a legacy boundary and is not solved by this release.
- `reference_clawdbot` is not touched.
