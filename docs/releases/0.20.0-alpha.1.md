## Summary

YonerAI v0.20.0-alpha.1 connects the public CLI to the staging YonerAIAPI
Control Spine. A logged-in staging CLI can inspect the staging account,
project/session state, API status, API rate-limit state, scopes, and sanitized
audit availability without receiving Google tokens.

This is staging-only. It is not production Google login, not production
Official Managed Cloud, not production Oracle/cloud runtime, and not automatic
local-to-cloud upload.

## What Users Can Run

Set the staging origin:

```powershell
$env:YONERAI_STAGING_AUTH_ORIGIN = "https://api-staging.yonerai.com"
```

Start staging login. The public CLI opens or prints the browser URL, polls the
CLI bridge when requested, and stores only the safe YonerAI staging session
claim after linking:

```powershell
yonerai login --bridge --open-browser --wait-linked --pretty --lang ja
```

Check who the staging CLI is linked as:

```powershell
yonerai whoami --pretty --lang ja
yonerai auth status --pretty --lang ja
yonerai auth session status --pretty --lang ja
```

Inspect the staging API Control Spine:

```powershell
yonerai api status --pretty --lang ja
yonerai api ping --pretty --lang ja
yonerai api rate-limit --pretty --lang ja
```

Inspect project and session state:

```powershell
yonerai project list --pretty --lang ja
yonerai project current --pretty --lang ja
yonerai auth sessions --pretty --lang ja
yonerai auth revoke-session <session_id> --pretty --lang ja
```

Inspect sanitized audit availability when the staging backend supports it:

```powershell
yonerai audit list --pretty --lang ja
```

Open the same areas from the interactive CLI:

```text
/認証
/ログイン
/API
/プロジェクト
/セッション
/監査
/同期
```

Clear the local staging session claim:

```powershell
yonerai auth logout --staging --pretty --lang ja
```

## Install / Repair Alpha

The stable one-command installer still installs the stable line by default:

```powershell
irm https://install.yonerai.com | iex
```

To install or repair this alpha explicitly from GitHub Release assets:

```powershell
& ([scriptblock]::Create((irm https://github.com/YoneRai12/YonerAI/releases/download/v0.20.0-alpha.1/install.ps1))) -Channel alpha -Version 0.20.0-alpha.1 -Repair -Execute -Launch
```

Add `-SetPath` only when you want the installer to create or refresh the user
PATH wrapper at `%LOCALAPPDATA%\YonerAI\bin\yonerai.cmd`.

## Verified Behavior

Local targeted validation covered:

- Control Spine status and rate-limit parsing.
- `yonerai login` staging alias without network by default.
- `yonerai whoami`, `yonerai project`, `yonerai auth sessions`, and
  `yonerai audit` fail closed when staging auth is missing.
- Unexpected Google token, refresh token, auth code, private endpoint, or local
  path fields from the backend are rejected or redacted before output.
- TUI slash commands remain Japanese-first while English aliases remain usable.
- PR #521's security finding is covered: linked auth state requires a validated
  `account/me` response and is not accepted from poll-only data.

Live staging smoke covered:

- `GET https://api-staging.yonerai.com/v1/status` through
  `yonerai api status --json`.
- `GET https://api-staging.yonerai.com/v1/rate-limit` through
  `yonerai api rate-limit --json`.
- unauthenticated `whoami` and `project list` return controlled
  `staging_auth_required` errors without printing local paths or tokens.
- `yonerai login --json` prints the staging browser URL without performing a
  token exchange in the public CLI.

## Security and Privacy Boundary

- staging origin remains allowlisted
- non-HTTPS public origins are rejected
- credentialed URLs and private IP origins are rejected
- redirects are rejected before following unexpected locations
- public CLI stores only a YonerAI staging session claim, not Google tokens
- pretty/JSON output prints session availability and hash only, not the value
- account email is redacted
- scopes are displayed as policy metadata, not authorization secrets
- project and session commands require staging auth
- session revocation sends only the selected session id to the staging backend
- audit output is sanitized metadata only
- local private memory, local file content, and local node content are not uploaded
- OpenAI shared traffic remains off
- production Google login remains unavailable

## Known Limitations

- This is an alpha prerelease line, not the stable CLI Local Runtime line.
- Production Google login/passkey/account runtime is not included.
- Production AWS backend is not included in the public repository.
- Production Oracle/cloud runtime is not included.
- Cloud conversation sync remains staging preview only.
- Local-to-cloud upload is not performed automatically.
- OpenAI shared traffic is not enabled.
- Local private memory is not uploaded automatically.
- Production cloud memory sync is not included.
- Production installer signing/trust store is not included.
- npm, winget, and auto-update apply are not included.
- Arbitrary shell/file/tool execution is not enabled.
- `src/cogs/ora.py` remains a legacy boundary and is not solved by this release.
- `reference_clawdbot` is not touched.
