# 2026.05.22 — YonerAI v0.1.0-alpha.2 Capability Slice

## Summary

YonerAI v0.1.0-alpha.2 is a local alpha CLI slice. It is useful for trying
mock provider execution, run IDs, workspace-scoped file summaries, mock search,
SafeShell planning, local-only memory, synthetic Discord gateway checks,
fixture-based status output, manifest verification, and installer dry-run
planning. It is not production-ready and it is not an installer-ready release.

Important traceability note: the tag `v0.1.0-alpha.2` points to PR #323. PRs
after that tag corrected the release notes, issue tracking, README guidance, and
installer dry-run command surface on `main`. The tag was not moved and no new
release was created.

## What changed since v0.1.0-alpha.1

- CLI diagnostics moved from a demo-only surface toward a runnable alpha CLI
  slice with `demo`, `doctor`, `status`, `plan`, `ask`, `search`, `ops`,
  `memory`, `discord synthetic`, `manifest verify`, and installer planning.
- Provider/planner execution spine now has task classification, provider/model
  planning, mock execution, run IDs, and run ledger/history.
- External provider adapters exist for OpenAI-compatible, Anthropic, and Gemini
  paths, but live calls require explicit opt-in.
- Local LLM execution is loopback-only.
- Workspace file summarization is explicit, allowlisted, text-only, and
  path-traversal resistant.
- Web search exists as a mock adapter by default.
- SafeShell exposes diagnostic planning only for a small allowlist.
- Local memory v0.1 is explicit opt-in, redacted, local-only storage.
- Discord gateway work is synthetic only.
- Official/cloud/oracle/status surfaces are contracts or fixtures, not
  production implementations.
- Installer work includes manifest schema/example, local manifest verification,
  artifact naming validation foundation, Windows dry-run planning, and the
  post-tag `yonerai install plan` dry-run command on `main`.

## YonerAI CLI

The useful public entry points are:

- `yonerai demo --pretty` and `yonerai demo --json`
- `yonerai ask ... --provider mock --json`
- `yonerai ask ... --file <path> --workspace <dir> --provider mock --json`
- `yonerai search mock ... --json`
- `yonerai ops plan git-status --json`
- `yonerai memory add/list/delete/export --store <local.jsonl> ...`
- `yonerai discord synthetic ... --json`
- `yonerai status --source fixture --json`
- `yonerai doctor --pretty`
- `yonerai manifest verify ...`
- `yonerai install plan --manifest ...` on current `main`
- `yonerai install plan-windows ...` as the Windows dry-run alias

Default CLI/demo paths do not require provider keys, Discord tokens, Oracle,
official cloud services, production databases, deployment, or live network
calls.

## Provider runtime

- Mock provider execution is the default safe path.
- OpenAI-compatible execution requires explicit live execution,
  provider-specific key/base-url environment configuration, and
  `YONERAI_OPENAI_COMPATIBLE_LIVE=1`.
- Anthropic execution requires explicit live execution, provider-specific key
  environment configuration, and `YONERAI_ANTHROPIC_LIVE=1`.
- Gemini execution requires explicit live execution, provider-specific key
  environment configuration, and `YONERAI_GEMINI_LIVE=1`.
- Local LLM execution rejects non-loopback endpoints.
- Default tests and default demo paths remain mock-only.

## Local files, search, and tools

- File summarization requires both `--file` and `--workspace`.
- The file path must stay inside the workspace allowlist.
- Mock search is deterministic and public-safe.
- Live web search is not default.
- SafeShell is a planner for known diagnostic operations; it is not arbitrary
  shell execution.

## Memory

Local memory v0.1 is explicit and local-only. It writes only when a store path
and confirmation flag are supplied. It is redacted before write and should not
be called complete persistent memory.

## Discord

Discord support in this public slice is synthetic. `yonerai discord synthetic`
checks adapter behavior such as progress events, final-once handling, and
duplicate responder prevention. It does not restore live Discord behavior.

## Installer and distribution

- `yonerai manifest verify` validates a local manifest file only.
- `yonerai install plan --manifest releases/manifest.example.json --json`
  validates a local manifest and prints a dry-run install plan.
- `yonerai install plan-windows --json` remains a Windows dry-run alias.
- The example manifest is contract-valid but not install-ready because it uses a
  non-production placeholder signature.
- Real installation still requires signed manifest verification, release asset
  hash verification, rollback behavior, safe uninstall docs, and a production
  signing/trust lifecycle.
- No network installer, `irm ... | iex`, npm package, winget package, production
  signing key, or production trust store is included.

## Security and boundary

- Provider keys are not required for default tests or default demos.
- Provider errors and local diagnostics must not print API keys.
- Live provider calls are opt-in only.
- Local LLM endpoints are loopback-only.
- Workspace file reads are explicit and bounded.
- Tool/search/shell surfaces are mock, disabled, or plan-only by default.
- Official Managed Cloud stays external contract-only in this public repo.
- Oracle/control-plane behavior stays contract/stub/local-dev only.
- No telemetry, real prompts, or real completions are ingested into analytics.

## Known limitations

- Not production-ready.
- Installer distribution is not ready.
- Official Managed Cloud is not runnable from this public repository.
- Oracle/control-plane behavior is not production.
- Live Discord is not restored.
- Persistent memory is not complete.
- Live web search is not default/live.
- SafeShell is not arbitrary shell.
- Workspace file summarization is not arbitrary filesystem access.
- npm and winget are not ready.
- No Google login.
- No production DB behavior.
- No production telemetry ingestion.
- No production signing keys or production trust stores.
- `src/cogs/ora.py` remains unresolved legacy/runtime boundary residue.
