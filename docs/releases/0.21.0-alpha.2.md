## Summary

YonerAI v0.21.0-alpha.2 checks the short staging Control Spine commands in
real use and fixes the update path so normal users do not need to remember long
flag sets.

This is a staging Control Spine prerelease only. It is not production Google
login, not production Official Managed Cloud, not production Oracle/cloud
runtime, and not automatic local-to-cloud upload.

## Commands

Set the staging origin when using the staging backend:

```powershell
$env:YONERAI_STAGING_AUTH_ORIGIN = "https://api-staging.yonerai.com"
```

Short user commands:

```powershell
yonerai login
yonerai whoami
yonerai sessions
yonerai revoke <session_id>
yonerai logout
yonerai projects
yonerai projects current
yonerai projects use <project_id>
yonerai ping
yonerai rate-limit
```

Short update commands:

```powershell
yonerai update
yonerai update stable
yonerai update alpha
```

`yonerai update` shows a two-step choice between the stable release channel and
the alpha prerelease channel. It does not auto-download, auto-install, mutate
PATH, run a remote script, or force an update.

Advanced/CI commands remain available:

```powershell
yonerai update check --manifest releases/manifest.v0.21.0-alpha.2.json --pretty
yonerai update plan --manifest releases/manifest.v0.21.0-alpha.2.json --pretty
```

## User-Visible Changes

- `yonerai update` now opens a safe choice screen instead of requiring users to
  remember `check`, `plan`, or manifest flags.
- `yonerai update stable` checks the stable channel.
- `yonerai update alpha` checks the alpha prerelease channel.
- The stable channel default now points at `v0.7.0`, not older `v0.6.x`
  installer text.
- `docs/site/yonerai.com/install.md`, README files, the Cloudflare short
  installer wrapper, and `src/web/static/install.txt` now describe the same
  stable install/update boundary.
- Stale `tests/test_official_bridge_foundation.py` expectations were updated
  from old stable manifest names to current stable/alpha truth.
- Live usage verification confirmed that short unauthenticated Control Spine commands return
  clear Japanese guidance and suggest `yonerai login` without stack traces.
- Live usage verification confirmed that `yonerai rate-limit` hides private operational metadata
  and only displays public-safe quota/scope information.
- Real-browser staging login E2E was run once with redacted evidence in
  `docs/dogfood/LOGIN_E2E_2026-06.md`.
- `docs/dogfood/DOGFOOD_LOG_2026-06.md` records the real-use check result and release
  blockers checked for this patch.

## Security And Privacy Boundary

- Production Google login remains disabled.
- `login --no-staging` remains rejected.
- Google access tokens, Google ID tokens, Google auth codes, and refresh tokens
  are not stored or printed by the public CLI.
- The standing token-custody regression remains part of the Quality Wall.
- OpenAI shared traffic remains disabled.
- Local private file, memory, and local node content are not uploaded
  automatically.
- No arbitrary shell/file/tool execution is enabled.
- No production Oracle/cloud runtime is included.
- No provider keys are included, printed, or stored.
- `agent:run` and `admin:*` scopes remain disabled by default and require a
  separate threat-model gate before any use.

## Known Limitations

- This is an alpha prerelease line, not the stable CLI Local Runtime line.
- Production Google login/passkey/account runtime is not included.
- Production AWS backend code is not included in the public repository.
- Production Oracle/cloud runtime is not included.
- Official Managed Cloud remains external/private and contract-only from the
  public repository.
- Cloud conversation sync remains staging preview only.
- Local-to-cloud upload is not performed automatically.
- Production installer signing/trust store is not included.
- npm, winget, and auto-update apply are not included.
- `src/cogs/ora.py` remains a legacy boundary and is not solved by this release.
- `reference_clawdbot` is not touched.
