## v0.9 Release Gate Conditions

A future `v0.9` release candidate is blocked until all of these are true:

- Owner explicitly authorizes release-candidate work, including any tag or
  GitHub Release action.
- Implementation lanes after `v0.8.0-alpha.1` land user-visible runnable
  changes; docs-only or checkpoint-only work stays in
  `docs/changelog/checkpoints/`.
- `VERSION` is updated only when the release gate is ready.
- The tag exactly matches `v<VERSION>`.
- The GitHub prerelease flag matches the version form. An alpha version such as
  `0.9.0-alpha.1` must be a GitHub prerelease.
- `docs/releases/<VERSION>.md` exists and is operation-manual style.
- `docs/RELEASE_NOTES.md` and `CHANGELOG.md` point to the same release facts
  without production or completion claims.
- `releases/manifest.v<VERSION>.json` exists and matches the schema.
- The final artifact name is versioned, for example `YonerAI-0.9.0-alpha.1.zip`
  if that version is selected.
- The manifest records the final artifact SHA256 and size.
- `production_ready` remains `false` unless a separately approved production
  lane proves otherwise.
- Placeholder or test trust is described as non-production. Do not claim
  production signing, production trust store, or signed-release trust unless
  the release includes a dedicated approved trust lane.
- Public runnable smoke validation is recorded.
- CLI smoke validation is recorded when CLI behavior is included.
- Web smoke validation is recorded when Web behavior is included.
- `git diff --check`, touched-path tests, release gate validation, secret/local
  path scan, and mojibake/hidden-Unicode scan pass for changed public text.
- No unresolved P0/P1/security/correctness review or `RELEASE_BLOCKER` marker
  remains.

## Version Rules

- Prefer semantic prerelease versioning for the next public milestone. If the
  owner confirms the v0.9 lane, the expected shape is likely
  `0.9.0-alpha.1`.
- Date-version GitHub Releases are frozen for routine checkpoints. Use a
  date-version only if the owner explicitly overrides release governance for a
  real runnable public milestone.
- Do not delete, retag, or rewrite historical releases.
- Do not create a GitHub Release for docs-only, process-only,
  checkpoint-only, ledger-only, root-inventory-only, or PR-count reconciliation
  work.

## Operation-Manual Release Note Requirements

A future `v0.9` release body should be written as an operation manual, not as a
marketing or completion claim. It should include:

- Summary.
- Runnable changes.
- Install / run instructions.
- A command table or equivalent that states what each command does, what it
  does not do, and its mode.
- Validation evidence with commands and results.
- Known limitations.
- Not included.
- Traceability to merged PRs, commits, tag, artifact, manifest, and any
  post-tag correction if one becomes necessary.
- Explicit installer/auth/privacy/self-evolution non-actions when those areas
  are included.
