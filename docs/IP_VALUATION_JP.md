## 1. 結論（エグゼクティブサマリー）

本リポジトリのIP価値は、**「コード資産 + アーキテクチャ設計 + 運用知見 + 拡張基盤」**の複合として評価するのが妥当です。

- **再調達原価（外注で同等品を再構築）ベースのレンジ**: **約 3,000万円〜1.5億円**
- 現状コード・機能群を中心にした「現実的な再現コスト（Current 相当）」: **約 5,000万円〜9,000万円**
- 商用SaaS水準まで含めた「運用・SRE・セキュリティ拡張込み」: **1億円超**も十分あり得る

> 補足: これは企業価値（株式価値）ではなく、技術資産の**再取得コスト / 代替コスト**に主眼を置く評価です。

## 3. 価値算定の方法

本レポートでは、公開情報のみで妥当性を担保しやすい **コストアプローチ（再調達原価法）**を主採用。

- 理由1: 売上・利益・継続率などの事業KPIが未提示
- 理由2: 比較可能な取引事例（完全同等）が希少
- 理由3: 実装規模・技術難度・運用要件から再現コストを推定しやすい

補助的に、IP価値の上振れ要因として以下を加味:

- 設計ドキュメント群の整備
- テスト群
- 拡張機構（skills/MCP/relay/core分離）

## 6. このシステムが「IP」になっている根拠

単なる行数ではなく、以下の**再利用可能な設計知**が蓄積されている点がIP価値の核です。

1. **分割統治の設計**
   - Bot / Admin API / Core / UI / Relay の責務分離
2. **拡張点の標準化**
   - Skills, Tools, MCP 連携
3. **安全性設計**
   - 承認・監査・ポリシー・アクセス制御の思想
4. **運用可能性**
   - テスト群とドキュメント群の存在
5. **ハイブリッド志向**
   - ローカルファーストからクラウド拡張までの移行設計

これらは、他社が短期間で模倣しづらい「暗黙知の形式知化」であり、IPとして評価できます。

## 7. 価格レンジを上げる要因 / 下げる要因

### 上振れ要因
- 実運用データ（DAU、継続率、課金、解約率）が強い
- セキュリティ監査済み・SLA運用済み
- 独自データセットや高品質プロンプト資産が整理済み
- 導入先のユースケース適合度が高い

### 下振れ要因
- OSS依存比率が高く差別化が弱い部分
- 特定個人の暗黙知依存（属人性）
- 環境依存（Windows固有、ローカル前提）で横展開が難しい場合
- 収益化導線が未整備

## 9. DCF（Discounted Cash Flow）法でのIP価値評価（投資家向け）

ここからは、事業KPIが揃っている前提での**投資家向け評価フォーマット**です。

### 9.1 DCFの基本式

- 企業価値（または事業価値）  
  \( EV = \sum_{t=1}^{n} \frac{FCF_t}{(1+WACC)^t} + \frac{TV}{(1+WACC)^n} \)
- 終価（永続成長モデル）  
  \( TV = \frac{FCF_{n+1}}{WACC-g} \)

### 9.2 YonerAI向けの推奨入力項目

- 売上: サブスク + 導入費 + 保守契約 + API従量
- 原価: GPU/推論費、インフラ費、サポート
- 固定費: 開発人件費、SRE、営業、管理
- 運転資本: 売掛・前受の増減
- CAPEX: サーバーやR&Dの資産化分（必要なら）

### 9.3 サンプル前提（検証用・仮定）

> 以下はモデル例であり、実績値ではありません。

- 評価期間: 5年
- WACC: 12%（レンジ 10〜16%）
- 永続成長率 g: 1.0%（レンジ 0〜2%）
- 初年度売上: 6,000万円
- 売上成長率: 年35% → 25% → 20% → 15% → 10%
- FCFマージン: 8% → 12% → 16% → 18% → 20%

### 9.4 サンプル結果（概算）

- ベースケースEV: **約 7,000万〜1.2億円**
- 強気ケースEV: **約 1.3億〜2.0億円**
- 弱気ケースEV: **約 3,000万〜7,000万円**

### 9.5 投資家説明で必須の感応度分析

- 軸1: WACC（10%, 12%, 14%, 16%）
- 軸2: FCFマージン（-5pt / base / +5pt）
- 軸3: 成長率（base ± 5pt）

この3軸をマトリクス化し、**評価レンジの妥当幅**として提示します。

## 10. ロイヤルティ免除法（Relief-from-Royalty）でのIP価値評価

SaaS/APIのIP評価で使いやすい手法です。

### 10.1 基本式

\[
IP\ Value = \sum_{t=1}^{n} \frac{Revenue_t \times Royalty\ Rate \times (1-Tax\ Rate)}{(1+r)^t}
\]

### 10.2 YonerAIでのロイヤルティ率の置き方（実務レンジ）

- OSSベース+限定的差別化: 2〜4%
- プロダクト差別化が明確: 4〜8%
- 高い独自性/参入障壁: 8〜12%

### 10.3 サンプル前提（検証用・仮定）

- 対象売上: 年6,000万円スタート、5年間成長
- ロイヤルティ率: 5%
- 実効税率: 30%
- 割引率 r: 12%

### 10.4 サンプル結果（概算）

- ロイヤルティ免除法のIP価値: **約 2,000万〜6,000万円**

> 備考: ロイヤルティ免除法は、IP単体価値を保守的に出す傾向があるため、コスト法やDCFと併用してレンジを組みます。

## 12. ライセンスの矛盾を解消する実務手順

このrepoでは、過去に**ライセンス表記の不整合**（`LICENSE` は CC BY-NC-ND 4.0 だがREADMEが MIT 表記）が起き得るため、以下の手順で継続的に整合を取ります。

### 12.1 まず決めるべき方針

方針A: **オープンソース配布重視**  
- 典型: MIT / Apache-2.0
- メリット: 採用されやすい、企業導入しやすい
- デメリット: コード再利用を広く許容

方針B: **商用コントロール重視**  
- 典型: ソース公開 + 商用利用制限（独自ライセンス）
- メリット: 商用利用を契約に寄せやすい
- デメリット: OSSエコシステムとの整合が難しい

> 注意: ソフトウェア本体にCC BY-NC-NDを使うと、改変再配布制限の扱いが厳しく、コントリビューションや企業利用で摩擦が出やすいです。

### 12.2 すぐやるチェックリスト

1. ルート`LICENSE`を最終方針に合わせて確定
2. `README.md` / `README_JP.md` のバッジ・文言を統一
3. `package.json` / `pyproject.toml` / 配布メタデータのlicense表記を統一
4. 依存OSSライセンス互換性を棚卸し（SBOM推奨）
5. `NOTICE` / `THIRD_PARTY_LICENSES` を作成
6. 外部提供時の利用規約（SaaS Terms）と整合

### 12.3 投資家向けに効く見せ方

- 「ライセンス方針」「商用利用可否」「二次利用条件」を1ページに明文化
- ライセンス不整合の解消履歴をリリースノート化
- 法務レビュー（弁護士）済みステータスを明記

## 13. 精度を上げるために追加で必要なデータ

より厳密な評価には、以下が必要です。

- 直近12か月の開発実績（実人月、工数ログ）
- 障害率、MTTR、運用負荷
- 利用指標（MAU/DAU、推論回数、継続率）
- 売上/案件化率（受託・SaaS・ライセンス）
- OSS依存と独自実装の比率

これらが揃えば、コスト法に加えてDCFやロイヤルティ免除法も適用可能になり、評価精度は大きく改善します。

- 本repoは、**PoCを超えた統合型AI基盤**として十分な規模と設計資産を持つ。
